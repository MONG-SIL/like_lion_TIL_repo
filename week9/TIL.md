# 📘 Today I Learned

### 1. 오늘 배운 내용

- 8주차까지는 아기 사자 명단을 React `useState`에만 저장해서, 새로고침하면 데이터가 사라졌다. 9주차에서는 **Supabase(PostgreSQL)** 에 저장해 브라우저를 닫거나 다른 기기에서 접속해도 같은 명단이 유지된다.
- Supabase는 DB·인증·실시간 구독을 제공하는 BaaS라, Express 같은 별도 백엔드 없이 프론트에서 `supabase.from('lions').select()` / `.insert()` / `.delete()`로 DB와 통신할 수 있다.
- 연결 정보는 `.env.local`의 `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`로 관리한다. URL은 대시보드 주소가 아니라 `https://<project-ref>.supabase.co` 형식의 **API URL**이어야 한다.
- `lib/supabase.ts`에서 `createClient<Database>`로 클라이언트를 만들고, `types/database.ts`에 DB 스키마 타입을 정의했다. DB는 `snake_case`, 앱은 `camelCase`라 `rowToLion` / `lionToRow` 변환 함수로 연결했다.
- **Row Level Security(RLS)** 로 `SELECT`는 누구나, `INSERT`/`DELETE`는 로그인 사용자만 허용했다. UI에서 버튼을 비활성화하는 것과 RLS는 역할이 다르고, anon key가 공개되므로 RLS가 실제 보안 역할을 한다.
- `useAuth` 훅에서 `getSession`과 `onAuthStateChange`로 로그인 상태를 유지하고, `/login` 페이지에서 이메일·비밀번호로 회원가입·로그인·로그아웃을 처리했다.
- `useLions` 훅은 앱 시작 시 Supabase에서 명단을 불러오고, `postgres_changes` 구독으로 추가·삭제를 실시간 반영한다. **조회는 로그인 여부와 무관**하고, 추가·삭제·랜덤 fetch는 로그인한 사용자만 가능하다.
- 라우팅은 `/`(목록), `/lions/:id`(상세), `/login`(로그인) SPA로 구성했고, 8주차의 URL 필터·정렬·검색, 요약/상세 카드, 비동기 상태 표시 기능은 그대로 유지했다.

### 2. 핵심 정리 (내 언어로)

- 데이터를 브라우저 state에만 두면 "내 화면에서만" 돌아가고, DB에 두면 **서비스처럼** 여러 기기·새로고침 후에도 유지된다.
- Supabase anon key는 프론트에 노출되므로, 버튼만 막아서는 부족하고 **RLS 정책**으로 수정 권한을 막아야 한다.
- "조회는 공개, 수정은 로그인"은 **UI(배너·비활성 버튼)** 와 **DB(RLS)** 를 함께 적용해야 완성된다.
- DB 컬럼명과 앱 타입이 다를 때 변환 함수를 한곳에 모아 두면, 8주차 TypeScript 구조와 자연스럽게 이어진다.
- 환경 변수 URL을 잘못 넣으면 `Failed to fetch`가 나므로, API URL과 Supabase 대시보드 URL을 구분해야 한다.

### 4. 느낀 점

- `.env.local`에 대시보드 URL을 넣어 `Failed to fetch`가 났을 때, API 엔드포인트와 관리 UI 주소가 다르다는 걸 직접 겪으며 배웠다.
- 8주차까지는 내 브라우저 안에서만 데이터가 움직였는데, Supabase를 붙이니 **같은 DB를 여러 환경이 공유**하는 느낌이 확실히 달랐다.
- 버튼 비활성만으로도 UX는 충분해 보이지만, RLS까지 설정해야 **진짜 권한 분리**가 된다는 점이 인상적이었다.
- `rowToLion` / `lionToRow`처럼 DB와 앱 사이에 변환 경계를 두니, 타입 안전성을 유지하면서 Supabase를 붙이기 수월했다.
